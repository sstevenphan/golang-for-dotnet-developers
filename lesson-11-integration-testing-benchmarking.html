Testing Strategy

Đừng chỉ test hàm nhỏ, hãy test cả luồng request thật

Vì Go code thường explicit và ít framework magic, integration test bằng HTTP test server tương đối nhẹ. Đây là cơ hội tốt để kiểm thử gần production hơn mà không cần harness quá đồ sộ. 

server := httptest.NewServer(mux)
defer server.Close()

resp, err := http.Get(server.URL + "/health")
if err != nil {
    t.Fatal(err)
}
Benchmarks

Go benchmark rất tiện để đo chi phí của hot path

func BenchmarkNormalizeEmail(b *testing.B) {
    for i := 0; i < b.N; i++ {
        NormalizeEmail(" Demo@Example.com ")
    }
}
  • Benchmark tốt khi bạn đã có giả thuyết về bottleneck.
  • Đừng benchmark cả hệ thống một cách mù mờ rồi rút kết luận lớn.
  • Ưu tiên benchmark hàm thuần, parser, serializer hoặc đoạn xử lý được gọi cực nhiều.
Race Detector

`go test -race` là công cụ rất nên bật với code concurrent

Với bài toán goroutine, shared state và cache, race detector có thể giúp bạn bắt bug mà code review rất dễ bỏ sót. Đây là một lợi thế lớn trong runtime/tooling của Go, đặc biệt khi service của bạn bắt đầu có background workers.

Thói quen tốt Mỗi khi chạm vào concurrency đáng kể, hãy chạy test với race detector ít nhất ở local hoặc CI cho vùng code đó.
Profiling

Chỉ tối ưu sau khi đã có profile hoặc số liệu đủ tin cậy

Công cụ Mục đích Khi dùng
Benchmark Đo throughput/latency của đoạn code nhỏ Khi nghi ngờ hot path cụ thể
Race detector Bắt data race Khi có shared mutable state hoặc goroutine phức tạp
Profiler Xem CPU, memory, allocation Khi service chậm hoặc tốn tài nguyên thật
