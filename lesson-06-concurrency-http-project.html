Mốc hoàn thành

Concurrency

Goroutine là đơn vị concurrent execution cực nhẹ

go func() {
    fmt.Println("running in background")
}()

Nếu bạn quen `Task.Run`, hãy xem goroutine như primitive nền thấp hơn và rẻ hơn. Điểm khác biệt là Go thiết kế cả ngôn ngữ và runtime để concurrency là chuyện bình thường, không phải lớp abstraction phụ thêm.

Channels

Channel giúp các goroutine giao tiếp với nhau một cách có cấu trúc

messages := make(chan string)

go func() {
    messages <- "done"
}()

msg := <-messages
fmt.Println(msg)

Bạn không phải lúc nào cũng cần channel. Nhiều trường hợp `sync.WaitGroup`, mutex hoặc worker pool đơn giản sẽ phù hợp hơn. Nhưng channel rất mạnh khi bạn muốn mô hình hóa dữ liệu chảy qua các stage xử lý.

Context

`context.Context` là thứ bạn phải dùng đúng ngay từ đầu

ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
defer cancel()

result, err := client.Fetch(ctx)
  • Dùng `context` để propagate timeout, cancellation, request-scoped data thật tiết chế.
  • Đừng cất `context` trong struct lâu dài.
  • Với HTTP service, `r.Context()` là gốc quan trọng để downstream operation biết khi request bị huỷ.
HTTP Service

Một service Go nhỏ có thể rất thẳng và dễ đọc

mux := http.NewServeMux()

mux.HandleFunc("GET /health", func(w http.ResponseWriter, r *http.Request) {
    w.WriteHeader(http.StatusOK)
    _, _ = w.Write([]byte("ok"))
})

server := &http.Server{
    Addr:    ":8080",
    Handler: mux,
}

log.Fatal(server.ListenAndServe())

Với .NET developer, đoạn này có thể khiến bạn ngạc nhiên vì mức độ “ít framework” của nó. Nhưng đó chính là điểm hay: phần cơ bản đã đủ dùng cho rất nhiều API nội bộ và service nhỏ. Khi cần nâng cấp, bạn mới bổ sung middleware, config, logging, routing nâng cao hoặc framework phụ trợ.

JSON

Go làm JSON mapping qua `encoding/json` và struct tag

type CreateUserRequest struct {
    Name  string `json:"name"`
    Email string `json:"email"`
}

Nếu bạn quen DTO trong ASP.NET Core, đây là ý tưởng tương tự nhưng tối giản hơn. Không có data annotation giàu tính năng mặc định như trong .NET; thay vào đó bạn thường tách rõ parse input, validate và business rules.

Binding

Parse request body một cách explicit và kiểm soát lỗi ngay tại handler

var req CreateUserRequest
if err := json.NewDecoder(r.Body).Decode(&req); err != nil {
    http.Error(w, "invalid json body", http.StatusBadRequest)
    return
}

Trong Go, parse và validate thường hiện ngay trong handler hoặc một lớp mỏng gần handler. Bạn sẽ ít thấy model binding magic hơn ASP.NET Core, nhưng đổi lại luồng dữ liệu cực kỳ rõ ràng.

Configuration

Tư duy config trong Go thường đi theo env-first và constructor rõ ràng

.NET Go Ý nghĩa
`appsettings.json` + Options Environment variables + config struct Gọn hơn cho container và deploy cloud-native.
Configuration providers phong phú Thường giữ config pipeline rất mỏng Ít framework magic, nhiều explicit setup. 
type Config struct {
    HTTPPort string
    DBConn   string
    LogLevel string
}

func LoadConfig() Config {
    return Config{
        HTTPPort: getEnv("HTTP_PORT", "8080"),
        DBConn:   getEnv("DB_CONN", ""),
        LogLevel: getEnv("LOG_LEVEL", "info"),
    }
}
Logging

Đừng log theo kiểu chuỗi tự do quá sớm, hãy nghĩ theo event có key-value

logger.Info("user created",
    "user_id", user.ID,
    "email", user.Email,
    "source", "http",
)
  • Structured logging giúp filter, search và correlate dễ hơn khi chạy thật.
  • Log nên gắn business context vừa đủ, không spam toàn bộ object.
  • Phân biệt rõ log cho operational event và log cho error path.
Rule of thumb Hãy log để người trực ca production hiểu chuyện gì xảy ra sau 20 giây đọc log, không phải để dev local “cảm thấy yên tâm”.

Database, SQL, transaction

SQL-first

Trong Go, biết SQL tốt thường quan trọng hơn biết một ORM kỳ diệu

Bạn vẫn có thể dùng ORM hoặc query builder, nhưng nền tảng của hệ sinh thái Go là `database/sql`. Nó cho bạn control rõ ràng hơn về query, connection pool, transaction và mapping dữ liệu.

Chuyển mental model: Từ “domain model dẫn dắt query” sang “query dẫn dắt shape dữ liệu mà use case cần”.
Query

Đọc dữ liệu bằng query explicit và scan vào struct nhỏ

row := db.QueryRowContext(ctx,
    `select id, name, email from users where id = $1`,
    id,
)

var user User
if err := row.Scan(&user.ID, &user.Name, &user.Email); err != nil {
    return User{}, err
}

Nếu so với EF Core, phần này ít “developer comfort” hơn nhưng lại rất minh bạch: bạn biết chính xác query gì chạy, cột nào được đọc, object nào được map, và chi phí truy cập nằm ở đâu.

Transactions

Transaction nên gắn với use case, không gắn mù quáng với từng repository call

tx, err := db.BeginTx(ctx, nil)
if err != nil {
    return err
}
defer tx.Rollback()

if err := ordersRepo.Insert(ctx, tx, order); err != nil {
    return err
}

if err := auditRepo.Insert(ctx, tx, auditEvent); err != nil {
    return err
}

return tx.Commit()
  • Transaction là concern của application flow hoặc use case, không chỉ của database helper.
  • Đừng mở transaction quá sớm rồi giữ quá lâu qua I/O ngoài database.
  • Nếu cần atomicity, hãy làm cho boundary đó thật lộ rõ trong code.
Abstraction

Logic persistence phức tạp Tách repository rõ hơn Dễ thay đổi và test theo capability.
Nhiều query lặp lại Dùng helper hoặc code generation Giảm boilerplate nhưng vẫn giữ SQL visible.

Project structure và dependency wiring

Đây là bài trả lời câu hỏi mà nhiều senior .NET developer sẽ hỏi rất sớm: “Nếu không có solution nhiều project và DI container mạnh tay, tôi tổ chức code thế nào cho sạch?”

Structure

Bắt đầu từ package theo capability, đừng tái tạo nguyên solution tree của .NET

/cmd/api
/internal/user
/internal/order
/internal/platform/db
/internal/platform/httpserver

Một project Go production tốt thường có ít tầng hình thức hơn bạn quen trong .NET. Điều quan trọng là dependency direction rõ, package name gọn, và người đọc nhìn vào là đoán được trách nhiệm.

Wiring

Constructor injection bằng tay thường đã đủ cho đa số service

userRepo := user.NewRepository(db)
userService := user.NewService(userRepo, logger)
userHandler := user.NewHandler(userService, logger)

Đúng là bạn mất tính “auto resolve” của DI container, nhưng bạn có được một composition root cực rõ. Điều này giúp debug dependency graph nhanh hơn và tránh framework coupling không cần thiết.

Architecture

Clean architecture trong Go nên là discipline, không phải nghi thức

  • Giữ boundary giữa handler, use case, persistence khi nó giúp thay đổi dễ hơn.
  • Đừng ép mọi package phải có interface chỉ vì template kiến trúc nói vậy.
  • Hãy tối ưu cho developer kế tiếp đọc code trong 5 phút, không phải cho sơ đồ đẹp trên slide.
Dấu hiệu over-engineering Service chỉ có vài use case nhưng đã có quá nhiều interface, factory, adapter, module registration và naming ceremony.

ASP.NET Core Go Trade-off

Middleware

Middleware là nơi gom concern cắt ngang, nhưng đừng nhồi business logic vào đó

func Logging(next http.Handler, logger *slog.Logger) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        start := time.Now()
        next.ServeHTTP(w, r)
        logger.Info("request completed",
            "method", r.Method,
            "path", r.URL.Path,
            "duration_ms", time.Since(start).Milliseconds(),
        )
    })
}

Hãy dùng middleware cho logging, tracing, auth gate, request ID, recovery và rate limiting. Còn business rule nên ở handler hoặc service layer, nếu không pipeline sẽ sớm khó đọc.

Authentication

Auth trong Go thường explicit hơn, ít “được framework lo sẵn” hơn ASP.NET Core

  • JWT, API key hoặc session đều có thể làm tốt, nhưng wiring sẽ rõ bằng code thay vì config-driven nhiều lớp.
  • Phân biệt rõ authentication và authorization, đừng gộp chung trong một middleware khổng lồ.
  • Khi auth phức tạp, hãy tách parser token, principal builder và policy check thành phần riêng.
Observability

Log thôi chưa đủ, production cần cả metrics và trace

Loại tín hiệu Dùng để làm gì Ví dụ
Logs Xem chi tiết event cụ thể Error context, payload summary, request lifecycle
Metrics Quan sát xu hướng hệ thống RPS, latency, error rate, queue depth
Traces Theo dõi request xuyên nhiều service Latency breakdown, dependency bottleneck
Nguyên tắc thực chiến Nếu service gọi database, cache, message broker hoặc downstream HTTP khác, trace sẽ giúp bạn tìm bottleneck nhanh hơn log rất nhiều.
Shutdown

Graceful shutdown là phần rất nên làm chuẩn từ sớm

ctx, stop := signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGTERM)
defer stop()

go func() {
    <-ctx.Done()
    shutdownCtx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
    defer cancel()
    _ = server.Shutdown(shutdownCtx)
}()

Trong môi trường container hoặc Kubernetes, đây không phải “nice to have”. Nó là phần giúp request đang chạy, connection đang mở và background job có cơ hội kết thúc đúng cách trước khi process chết.

Testing Strategy

Benchmarks

Go benchmark rất tiện để đo chi phí của hot path

func BenchmarkNormalizeEmail(b *testing.B) {
    for i := 0; i < b.N; i++ {
        NormalizeEmail(" Demo@Example.com ")
    }
}
  • Benchmark tốt khi bạn đã có giả thuyết về bottleneck.
  • Đừng benchmark cả hệ thống một cách mù mờ rồi rút kết luận lớn.
  • Ưu tiên benchmark hàm thuần, parser, serializer hoặc đoạn xử lý được gọi cực nhiều.
Race Detector

`go test -race` là công cụ rất nên bật với code concurrent

Với bài toán goroutine, shared state và cache, race detector có thể giúp bạn bắt bug mà code review rất dễ bỏ sót. Đây là một lợi thế lớn trong runtime/tooling của Go, đặc biệt khi service của bạn bắt đầu có background workers.

Thói quen tốt Mỗi khi chạm vào concurrency đáng kể, hãy chạy test với race detector ít nhất ở local hoặc CI cho vùng code đó.

Performance

Việc Go nhanh không có nghĩa mọi code Go đều tự động hiệu quả. Allocation thừa, query dở, log quá nặng, goroutine vô tội vạ hoặc contention cao vẫn khiến hệ thống xuống sức như thường.
