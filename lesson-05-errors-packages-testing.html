Error handling, package, testing

Mốc hoàn thành

Errors

Go không giấu luồng lỗi sau exception mặc định

user, err := repo.GetByID(id)
if err != nil {
    return fmt.Errorf("load user %d: %w", id, err)
}

Với người mới từ C#, đoạn `if err != nil` lặp lại có thể trông hơi dài. Nhưng bù lại, luồng lỗi hiện ra ngay trên mặt code. Ở service và infra layer, đây là một lợi thế lớn cho việc đọc trace logic.

Wrapping

Hãy wrap lỗi để thêm ngữ cảnh, nhưng đừng làm mất lỗi gốc

if err != nil {
    return fmt.Errorf("save order: %w", err)
}

if errors.Is(err, sql.ErrNoRows) {
    return nil, ErrUserNotFound
}

Cách nghĩ tương đương trong .NET sẽ gần với việc preserve inner exception và thêm message context, nhưng Go làm điều này theo hướng đơn giản hơn và rất dễ scan bằng mắt.

Packages

Tổ chức package theo boundary thực dụng, không cần tái hiện solution tree của .NET

  • Bắt đầu nhỏ: package theo domain hoặc capability thay vì chia lớp n-level quá sớm.
  • Tránh tạo package chỉ để chứa một abstraction chưa có người dùng rõ ràng.
  • Ưu tiên dependency direction đơn giản, import graph dễ nhìn, tránh vòng lặp package.
Gợi ý cho người từ clean architecture Bạn vẫn có thể giữ boundary rõ ràng, nhưng nên làm bằng package gọn và constructor explicit thay vì dựng framework nội bộ sớm.
Testing

Table-driven test là idiom rất phổ biến

func TestAdd(t *testing.T) {
    tests := []struct {
        name string
        a    int
        b    int
        want int
    }{
        {name: "positive numbers", a: 2, b: 3, want: 5},
        {name: "zero", a: 0, b: 0, want: 0},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            got := Add(tt.a, tt.b)
            if got != tt.want {
                t.Fatalf("want %d, got %d", tt.want, got)
            }
        })
    }
}

Nếu bạn quen xUnit/NUnit + `[Theory]`, thì table-driven test là phiên bản Go-idiomatic của tư duy đó. Nó đơn giản, rõ data setup, và scale khá tốt cho logic thuần.

Takeaway

3 nguyên tắc nên giữ khi viết Go production

  • Error phải rõ ngữ cảnh, đừng trả lỗi trần trụi khi đi qua nhiều lớp.
  • Package tree nên tối ưu cho khả năng đọc và dependency direction, không phải cho “độ đẹp lý thuyết”.
  • Test nhỏ, trực diện, data-driven thường phù hợp hơn những test harness quá nặng.
